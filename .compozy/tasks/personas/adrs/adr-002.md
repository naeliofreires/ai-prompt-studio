# ADR-002: Local Promptizer View with Seeded Editable Personas

## Status

Accepted

## Date

2026-07-07

## Context

Promptizer is currently a single React screen composed through `PromptizerApp`, `PromptStudioScreen`, `usePromptStudioController`, `useRoles`, and `usePromptGeneration`. Persona CRUD already exists for custom personas through the UI client, IPC contracts, and Electron-store/localStorage persistence. Built-in personas are currently immutable and sourced from `apps/promptizer/spec/personas.json`, which conflicts with the PRD requirement that the two initial personas be editable and deletable.

## Decision

Implement the Personas page as a local Promptizer view switch instead of adding a router. Seed two editable personas, Frontend Specialist and Backend Specialist, into the editable persona store. The Personas page will use inline create/edit forms with required trimmed title and description fields, confirmation before deletion, and existing persona CRUD plumbing where possible. Generate will remain guarded by the selected persona state and will expose clear UI guidance when no persona exists or no persona is selected.

## Alternatives Considered

### Alternative 1: Add an Internal Router

- **Description**: Introduce route-based navigation inside Promptizer for Studio and Personas.
- **Pros**: Scales to more pages later; URL-like mental model.
- **Cons**: Adds new routing infrastructure for a two-view MVP; increases integration and test surface.
- **Why rejected**: A local view switch satisfies the MVP with less architectural change.

### Alternative 2: Built-in Overrides Layer

- **Description**: Keep built-in personas immutable and add override/delete metadata on top of them.
- **Pros**: Preserves existing built-in source structure; can support reset-to-default later.
- **Cons**: Adds more state rules and edge cases than needed; delete/edit semantics become less direct.
- **Why rejected**: Seeded editable personas provide simpler user semantics and simpler CRUD behavior for the MVP.

### Alternative 3: Reuse Existing Modals

- **Description**: Keep current create/edit/delete modals and trigger them from the new page.
- **Pros**: Reuses existing UI code; fastest implementation path.
- **Cons**: Does not match the approved inline management experience; keeps management split across transient dialogs.
- **Why rejected**: The approved technical direction is inline create/edit forms on the Personas page.

## Consequences

### Positive

- Avoids introducing routing dependencies or global navigation changes.
- Keeps persona persistence and generate integration close to existing code paths.
- Makes the two initial personas truly editable and deletable.
- Keeps validation simple and consistent with the MVP field model.

### Negative

- Future Promptizer pages may require revisiting navigation structure.
- Seed logic must avoid duplicating personas across app launches.
- Moving away from immutable built-ins may require adapting existing tests and assumptions.

### Risks

- Existing built-in persona resolution may conflict with seeded editable personas. Mitigate by defining a clear source of truth for MVP personas during implementation.
- Deleting the active persona can leave generation blocked. Mitigate by updating selection state and showing clear empty/blocked UI.
- Inline editing can introduce unsaved-state complexity. Mitigate with one active form at a time and explicit save/cancel actions.

## Implementation Notes

- Keep the persona domain shape as `id`, `label`, and `role`; map UI title to `label` and UI description to `role`.
- Seed Frontend Specialist and Backend Specialist once into the editable store.
- Use local Promptizer state for `studio` versus `personas` view.
- Add tests for seeding, inline CRUD, delete confirmation, selection updates, and blocked generation with no personas.
