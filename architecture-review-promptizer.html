Collapse the API key management module

Strong in-process
ui/store/api-key-store.ts
ui/config/api-key-storage.ts
ui/api/api-key-sync-client.ts
ui/hooks/useApiKeySettings.ts
main/services/api-key-manager.ts
shared/domain/api-keys.ts

Before — 6 modules, 2 seams

After — 1 deep module, 1 seam

Problem: Six modules across two layers for one concept — API keys. Fire-and-forget sync swallows errors. Hydration races between store init and IPC. The Zustand store is a thin proxy over localStorage with no leverage.

Solution: Collapse into one deep module — `ApiKeyRepository` — that owns persistence, hydration, validation, masking, and IPC sync behind a single interface. The seam stays at the IPC boundary (one adapter for Electron, one for browser localStorage).

  • locality: key lifecycle bugs concentrate in one module
  • leverage: callers learn one interface for all key operations
  • tests: swap IPC adapter for in-memory; assert on observable state
  • delete Zustand store, storage config, sync client, settings hook

Problem: `register-handlers.ts` creates `LLMAdapter` and `PromptEvaluator` at module scope, baking the real `generateText` (Vercel AI SDK) into every IPC handler. No seam to replace them — tests must mock entire modules or skip the layer entirely.

Solution: Accept the adapters as parameters in `registerIpcHandlers(options)`. Production wiring happens at the Electron entry point; tests inject fakes. The existing `LlmAdapter` and `PromptEvaluator` interfaces already define the seam — just stop hard-coding inside.

  • locality: IPC handler logic testable without Vercel AI SDK
  • leverage: existing LlmAdapter seam becomes real (two adapters)
  • tests: inject in-memory adapter; no module mocking
  • wiring moves to the Electron entry point — composition responsibility

Consolidate the provider registry

Worth exploring in-process
spec/providers.json
shared/domain/provider.ts
shared/domain/provider-meta.ts
main/services/api-key-manager.ts
main/utils/resolve-language-model.ts

Before — 4 files for one concept

After — 1 registry module

Problem: Provider knowledge is scattered across 4 files with a 99-line switch statement. Adding a new provider requires coordinated edits in `providers.json`, `api-key-manager.ts` (env var), `provider-meta.ts` (placeholder), and `resolve-language-model.ts` (AI SDK mapping). No single place to reason about "what does this provider need?"

Solution: Consolidate into one `ProviderRegistry` module that owns the provider spec, metadata, key resolution, and model resolution. The spec JSON becomes a private detail; the interface is `resolveModel()` and `getProviderMeta()`.

  • locality: new provider = one place to edit
  • leverage: callers don't know about env vars, switches, or AI SDK internals
  • tests: register a mock provider, assert on model resolution
  • delete 99-line switch, scattered env var mapping

Remove orphaned modules and unexercised store

Speculative in-process
ui/hooks/usePromtizer.ts — unused hook
ui/hooks/useStreamedMarkdown.ts — unused hook
main/store/prompt-sessions-store.ts — no consumer
ui/services/promtizer.ts — partially superseded

Problem: Three modules have no consumers. `usePromtizer` was superseded by `usePromptGeneration`. `useStreamedMarkdown` was built for streaming output that was never wired. `prompt-sessions-store` saves sessions but no UI reads them. They fail the deletion test: deleting all three removes complexity with zero spread.

Solution: Delete them. If streaming or session history is needed later, the domain types in `shared/` survive. The seam at `LlmAdapter` still supports streaming if the output channel changes.

  • locality: no dead code to confuse newcomers
  • removes ~200 lines with zero risk

Top recommendation

Open the IPC handler composition root for injection

This is the highest-leverage change. The IPC handlers are the composition root of the entire main process — every main-process feature flows through them. Right now they are untestable without module mocking because adapters are baked at module scope. Opening this seam gives you testability across the entire main-process surface (prompt generation, evaluation, session saving, configuration management, key management) with a single structural change. It also unblocks meaningful tests for generate-refined-prompt.ts, which is already well-structured with injected dependencies but can't be exercised through its actual call path. Two adapters (production Vercel AI SDK, in-memory test fake) justify the seam.
